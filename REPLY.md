In general, my approach was to treat this as a data reconciliation problem first, and then a pipeline problem second. For the data reconciliation piece, I imagined merging two contact lists, where you may have the same people spelled differently and fields conflicting. With this, you need rules for "who is the same person," rules for "which value wins," and a record of every judgment call. In this case, each source gets its own adapter that can convert raw files (JSON from the API, and CSV from RealGM) into a standard record shape, a mapping table that ties each source's player identifiers to internal IDs I own, and a per-field priority config that decides which source wins when there's a disagreement. I did notice that RealGM tends to own the box score stats that the API doesn't have, the API owns advanced metrics that RealGM doesn't have, and for overlapping fields, I have the API win with RealGM as a fallback. Nothing ever gets silently dropped; if a row fails validation, it lands in the rejections table with the raw payload, and every cross-source disagreement is logged with both values and the winner.

I probably spent the most time on the messy edge cases, which included name normalization (including a character that standard accent stripping mishandles), minutes arriving in both decimal and MM:SS formats, and making the whole pipeline idempotent so that reruns don't alter existing data. The tests trace all 71 CSV rows to an outcome (loaded, skipped as duplicate, or logged) and verify TS% against an independent recomputation. I also added the bonus pieces: a replayable live game feed with a staleness guard and a small Django read API, all runnable with Docker.

With more time, I'd wire up the adapters into a real scheduler, as they're already shaped as job bodies. I'd also add a watermark table so freshness alerting is one query, land raw payloads in staging tables so backfills can replay history through new merge rules, and swap schema.sql for proper migrations. A few weaknesses come to mind: the RealGM matching is name-only, so two players sharing a name would merge (the fix would be corroborating attributes like date of birth, height and weight). Seasons are keyed on a player and season, so one row can blend two sources that disagree on a team, where we would want to add a "stint" field to the schema. Furthermore, a source can never retract data, since the upserts keep old values when a new run supplies NULL.
